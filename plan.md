# 论坛项目 0-1 上线完整计划书

## 1. 项目目标

项目名称：公开注册论坛平台  
项目阶段：0-1 搭建与首发上线  
目标结果：

1. 论坛可通过公网域名访问
2. 用户可以注册、登录、找回密码
3. 用户可以浏览板块、发帖、回复、上传头像
4. 管理员可以审核内容、封禁用户、处理举报
5. 网站具备 HTTPS、备份、基础防刷、防垃圾注册能力
6. 上线后支持持续运营和扩容

---

## 2. 当前基础条件

已具备条件：

1. 腾讯云轻量应用服务器
2. 宝塔面板已安装
3. 服务器公网 IP 已可访问
4. 已具备基础运维入口

建议服务器基线：

1. CPU：2 核及以上
2. 内存：4GB 及以上
3. 磁盘：40GB 及以上
4. 系统：Ubuntu 22.04 LTS 优先
5. 网络：已开放 80、443、22，后续仅保留必要端口

---

## 3. 论坛技术方案选型

## 3.1 候选方案对比

### 方案 A：Flarum
优点：

1. 轻量
2. 界面现代
3. 对 2C4G 服务器友好
4. PHP + MySQL 生态和宝塔兼容性好
5. 安装和维护成本相对低

缺点：

1. 原生大型社区能力不如 Discourse 完整
2. 插件生态需注意兼容性
3. 高并发能力不如更重型方案

适合：

1. 从 0 开始
2. 中小规模论坛
3. 预算有限
4. 希望先快速上线验证社区

### 方案 B：Discourse
优点：

1. 社区功能成熟
2. 邮件通知、审核、权限体系强
3. 大型论坛经验成熟

缺点：

1. 吃资源
2. 安装依赖 Docker
3. 运维复杂度更高
4. 对轻量服务器压力更大

适合：

1. 对论坛运营很熟
2. 预期用户量较大
3. 能接受较高运维复杂度

### 方案 C：NodeBB
优点：

1. 实时互动体验好
2. 现代论坛体验不错

缺点：

1. 依赖 Node.js 和 Redis
2. 运维复杂度高于 Flarum
3. 宝塔环境下也能做，但不如 PHP 方案省心

适合：

1. 有 Node.js 运维经验
2. 想做实时消息型社区

---

## 4. 推荐方案

推荐：Flarum

推荐原因：

1. 你当前已经在用宝塔
2. 轻量云服务器更适合先走轻量栈
3. Flarum 更适合低成本快速上线
4. 先把社区跑起来，比一开始追求重型架构更重要

推荐技术栈：

1. Nginx
2. PHP 8.2
3. MySQL 8.0 或 MariaDB 10.6+
4. Redis
5. Flarum
6. HTTPS 证书
7. SMTP 邮件服务
8. 对象存储可作为后续扩展项

## 6. 整体架构

用户访问路径：

1. 用户浏览器
2. 域名解析到腾讯云轻量服务器
3. Nginx 处理 HTTPS
4. Nginx 转发到 Flarum
5. Flarum 使用 PHP-FPM
6. 数据写入 MySQL
7. 缓存和队列用 Redis
8. 邮件通过 SMTP 服务发出

架构组成：

1. Web 层：Nginx
2. 应用层：PHP-FPM + Flarum
3. 数据层：MySQL
4. 缓存层：Redis
5. 文件层：本地磁盘，后续可切 COS
6. 安全层：HTTPS、防火墙、验证码、限流、登录保护
7. 运维层：宝塔、备份、监控、日志

## 7. 域名与访问规划

建议准备：

1. 一个主域名，例如 example.com
2. 一个论坛子域名，例如 forum.example.com
3. 一个管理员备用入口，仅自己知道
4. 域名 DNS 托管建议用腾讯云 DNSPod

推荐访问地址：

1. 主论坛地址：https://forum.example.com
2. 管理后台：使用主地址登录后台，不额外暴露新入口
3. 静态资源：前期与主站共用，后期可拆 CDN

域名规划建议：

1. 主站和论坛分开更清晰
2. 后续如果有官网，可用 www.example.com
3. 论坛单独用 forum.example.com，迁移更方便

---

## 8. 安全规划

必须项：

1. 只开放 22、80、443
2. 宝塔管理口不长期暴露给所有人
3. 修改宝塔默认安全入口
4. 修改宝塔管理员密码
5. 启用 HTTPS
6. 开启数据库强密码
7. 关闭不必要服务和端口
8. 定期备份网站和数据库
9. 设置 SSH 密钥登录，尽量禁用密码登录
10. 限制 root 直接远程登录
11. 配置基础防爆破
12. 配置 fail2ban 或同类策略

论坛业务安全：

1. 注册需邮箱验证
2. 开启验证码
3. 新用户首帖审核
4. 限制短时间重复发帖
5. 限制链接数量
6. 敏感词过滤
7. 举报机制
8. 黑名单 IP 和 UA 处理
9. 管理员操作日志保留

---

## 9. 邮件系统规划

论坛必须有邮件能力，否则注册体验会断掉。

邮件用途：

1. 注册验证
2. 找回密码
3. 系统通知
4. 举报处理通知
5. 管理员告警

推荐方案：

1. Mailgun
2. Resend
3. SendGrid
4. 腾讯企业邮箱 SMTP
5. 阿里云企业邮箱 SMTP

要求：

1. 使用真实发信域名
2. 配置 SPF
3. 配置 DKIM
4. 配置 DMARC
5. 避免进入垃圾箱

## 10. 用户与权限设计

用户角色建议：

1. 游客
2. 普通会员
3. 新注册用户
4. 认证用户
5. 版主
6. 超级管理员

权限基线：

4. 邮箱验证后才允许发言
5. 新注册 24 小时内限制外链和私信

## 11. 板块结构设计

建议初始只做 5 到 8 个板块，不要一开始太多。

示例结构：

1. 公告区
2. 新手报道
3. 综合讨论
4. 问题求助
5. 资源分享
6. 建议反馈
7. 站务处理
8. 违规举报

设计原则：

1. 板块少而清晰
2. 每个板块有明确主题说明
3. 先观察用户行为，再扩板块
4. 不要用过细分类导致冷启动空板块

## 13. 数据库与存储规划

数据库建议：

1. MySQL 8.0 优先
2. 字符集统一 utf8mb4
3. 每日自动备份
4. 备份文件异地保存

存储建议：

1. 前期头像和附件存在本地磁盘
2. 磁盘增长到一定规模后迁移对象存储
3. 大附件限制上传大小
4. 图片做压缩和格式限制

附件策略：

1. 图片：jpg、png、webp
2. 禁止直接上传危险脚本文件
3. 单文件大小限制
4. 总容量增长阈值监控

## 18. 风险清单

技术风险：

1. 插件兼容冲突
2. PHP 版本不兼容
3. 数据库字符集不统一
4. Redis 未正确启用
5. 备份未真正恢复验证

安全风险：

1. 弱密码
2. 暴露管理端口
3. 垃圾注册
4. 恶意灌水
5. 上传漏洞
6. SQL 注入或插件漏洞

业务风险：

1. 板块过多导致空城
2. 没有首批内容
3. 没有版主管理
4. 注册门槛过低导致垃圾用户泛滥
5. 注册门槛过高导致冷启动失败

应对措施：

1. 上线前最小化插件数量
2. 强制邮件验证
3. 开启验证码
4. 做每日备份
5. 做首批种子内容
6. 先安排版主制度

---

## 19. 数据备份与恢复计划

备份策略：

1. 数据库每日全量备份
2. 网站目录每日备份
3. 附件目录每日备份
4. 备份至少保留 7 到 30 天
5. 备份至少保留一份异地副本

恢复演练：

1. 每月至少做一次恢复演练
2. 验证数据库是否可恢复
3. 验证附件是否可恢复
4. 验证论坛是否能重新启动

恢复目标：

1. 关键数据不丢
2. 故障后可在数小时内恢复
